##### Пример 2

Библиотека для взаимодействия с геосервисом GraphHopper предоставляет ряд абстракций, отображающих HTTP-запросы и ответы в удобные структуры, 
которые позволяют работать с сервисом не заботясь о его внутреннем устройстве и мелких деталях по взаимодействию с ним.
Также множество HTTP-запросов отображается в ограниченный набор, который актуален для сервиса геокодирования. Мы поднимаемся на новый уровень, где вообще нет протокола HTTP.
Если я не ошибаюсь, то библиотеку graphHopper можно назвать абстракцией, которая отображает HTTP-клиент в набор инструментов для взаимодействия с геосервисом.

```java
public static Point getPointByAddress(String address) {
        //Сетевой запрос к сервису геокодирования представлен в виде структуры GHGeocodingRequest
        GHGeocodingRequest request = new GHGeocodingRequest(
                address,
                "",
                1);

        //Сетевой ответ сервиса геокодирования представлен в виде структуры GHGeocodingResponse
        GHGeocodingResponse response = ghGeocoding.geocode(request);
        GeometryFactory geometryFactory = new GeometryFactory();

        if (response.getHits().isEmpty()) {
            return geometryFactory.createPoint(new Coordinate(0.0, 0.0));
        }

        double lat = response.getHits().getFirst().getPoint().getLat();
        double lng = response.getHits().getFirst().getPoint().getLng();

        return geometryFactory.createPoint(new Coordinate(lng, lat));
    }
```

##### Пример 3

Отображение инструментов(драйвер) по работе с PostgreSQL в JDBC является абстракцией. 
JDBC избавляет нас от необходимости думать о деталях взаимодействия с той или иной БД. 
Эти детали взаимодействия отображаются на простые методы, которые одинаковы для разных БД.

```java
public <T> T executeQueryAndExtract(ResultSetExtractor<T> extractor, String query,
                                        Collection<? extends ParameterBinding> bindings) throws Exception {
        initConnectionFromPool();
        checkState();

        try (NamedParameterPreparedStatement npps = new NamedParameterPreparedStatement(conn, query)) {
            for (ParameterBinding binding : bindings) {
                binding.bindSelfNamed(npps);
            }

            if (logger.isDebugEnabled()) {
                logQuery(query, bindings);
            }

            //Выполняем запрос к БД не заботясь о деталях взаимодействия с конкретным вариантом базы.
            try (ResultSet rs = npps.executeQuery()) {
                return extractor.extract(rs);
            }
        }
    }
```

##### Пример 4

События сервиса отображаются в различные метрики. В процессе работы сервиса происходят разные вещи, но мы их отображаем на числа.
С числами проще взаимодействовать, так как мы сосредотачиваемся на важных для нас, в определённом контексте, параметрах.

```java
    public List<String> getAndDeleteSan() {
        //У нас имеется meterRegistry, который абстрагирует детали взаимодействия с конкретной системой мониторинга.
        List<String> deleted = meterRegistry.timer("nsq.deduplication.queue.delete")
                .record(deduplicator::getAndDeleteSan);
        //Удалённые учётные записи из таблицы дедупликации отображаются в число
        meterRegistry.counter("nsq.deduplication.queue.delete.total").increment(deleted.size());
        return deleted;
    }
```

##### Пример 5

Отображаем географические координаты в адрес. Человеку удобнее взаимодействовать с классическим адресом, нежели с числами широты и долготы.
Также небольшое отклонение в широте/долготе может вернуть тот же самый адрес. Это похоже на отображение напряжений транзисторов в числа, 
когда напряжение может немного плавать, но итоговое число не изменится.

```java
//Тут я бы назвал абстракцией саму функцию, которая отображает координаты в адрес.
public static Address getAddressByPoint(Point point) {
        GHGeocodingRequest request = new GHGeocodingRequest(
                GHPoint.create(point),
                "",
                1);

        GHGeocodingResponse response = ghGeocoding.geocode(request);

        if (response.getHits().isEmpty()) {
            return "Не удалось определить адрес";
        }

        GHGeocodingEntry addressInfo = response.getHits().getFirst();

        return Address.builder()
                .country(addressInfo.getCountry())
                .state(addressInfo.getState())
                .city(addressInfo.getCity())
                .street(addressInfo.getStreet())
                .houseNumber(addressInfo.getHouseNumber())
                .name(addressInfo.getName())
                .build();
    }
```

#### Определение Дейкстры
"Мы хотим открыть для себя новый семантический уровень мышления, на котором мы парадоксально можем быть абсолютно точны -- ещё более точны, чем даже на уровне синтаксиса кода."  
Кажется, что здесь имеется в виду следующее: нам нужно находить такие абстракции, которые позволят откинуть лишние детали и сосредоточиться на важных для нас вещах.
Таким образом, чтобы чётко увидеть какие-то свойства той или иной сущности, которую мы пытаемся отобразить. 
При размышлениях об этом определении мне пришло в голову выражение "За деревьями леса не видеть". Очень кажется, что оно сюда подходит.
А значит в работе нужно искать точные и обоснованные абстракции, которые позволят выйти на более высокий уровень думания о программе, 
что снизит когнитивную нагрузку при разбирательствах с кодом.

#### Вывод
Абстракции(отображения) можно сделать какие угодно и для чего угодно. Но они могут быть плохими: неточными и/или необоснованными. От плохих абстракций мало пользы.
А хорошие абстракции позволяют более продуктивно думать о какой-либо сущности и более продуктивно с ней работать. Хорошие абстракции позволяют предсказывать поведение моделируемой системы.
Таких абстракций можно найти множество. Один кусок кода может быть связан(ассоциирован) с множеством абстракций. То есть абстракция это сущность, которая явно не представлена в коде.

Я долгое время не мог понять что такое абстракция и даже не уверен, что сейчас правильно её понимаю. Но по мере размышления над текстом заметки я 
отсеял некоторые варианты и сделал определённые выводы. Возможно, чуть позже я пойму, что они не совсем верные. Чувствую, что нужно ещё подумать над этой темой и, наверное, ещё перечитать материал.

На данный момент выводы составил такие:
Абстракция не так проста, как о ней говорят и как её воспринимают. Многие вещи, которые подразумевают под абстракцией, так или иначе с ней связаны, но они не являются абстракцией.
Что такое абстракция? Это отображение одной сущности на другую. Но что такое отображение? Перевод, набор правил или это просто слово, которое говорит о том, что одна сущность представлена в другом виде?
В материале сказано, что абстракция никак не выглядит в коде и ещё там упоминается их бесплотная природа. Думаю, что абстракцией можно назвать способ(набор правил), которым мы отражаем одну сущность в другую.   
Хорошая абстракция должна быть обоснованной и точной.
Можно отражать множество одних вещей во множество других вещей, но нужно задаваться вопросом "Какие операции отображения можно точно использовать обращаясь только к абстрактной версии?". 
В статье Патрика и Радхии Кузо нашёл формулировку, которая натолкнула меня на следующее определение точности абстракции: "Абстракция точная, если результаты вычисления над абстрактными сущностями дают верную информацию о фактических вычислениях".
Если брать пример с телефонами из заметки, то абстракция с серийным номером будет точной, если его увеличение всегда будет соответствовать повышению мощности телефона. 
Хотя, будет вернее сказать, что абстракция будет точной лишь в этом конкретном случае. А вот точность прочих операций остаётся под сомнением.
