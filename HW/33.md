Допустим у нас есть объект представляющий описание Региона, пускай он называется Location. Мы хотим получать информацию о регионах в отчётах.
Но в отчёте должно использоваться особое имя региона, специфичное для отчётов. Мы бы хотели добавить в класс региона новое поле `reportLocationName`. 
Но он закрыт дя изменений. Поэтому реализуем добавление нового поля через словарь.

```java
class Location {
    private long id;
    private String name;
    //Прочие поля и методы...
}

final class ExtendedLocation {
    private static final Map<Location, String> locationReportNamesMap = new WeakHashMap<>();

    private ExtendedLocation() { }

    public static void setNameForReport(Location location, String nameInReport) {
        locationReportNamesMap.put(location, nameInReport);
    }

    public static String getNameForReport(Location location) {
        locationReportNamesMap.get(location);
    }
}

public static void main(String[] args) {
    Location loc = new Location(1, "Moscow");

    //Самый простой вариант
    Map<Location, String> locationReportNamesMap = new WeakHashMap<>();
    locationReportNamesMap.put(loc, "MOS");
    System.out.println(locationReportNamesMap.get(loc));
    
    //Чуть более сложный, но элегантный вариант
    ExtendedLocation.setNameForReport(loc, "MOS");
    System.out.println(ExtendedLocation.getNameForReport(loc));
}

```

#### Вывод
Мне кажется, что модель данных стала сложнее, потому что информация, относящаяся к одному объекту теперь хранится в разных частях кода. А это лишняя когнитивная нагрузка. 
Можно забыть об этом и что-то упустить.  
Если бы мы добавили поле в класс, то его можно было бы сделать иммутабельным(не создавать сеттеры), кажется, что со Словарём этот трюк провернуть сложнее и у нас может быть такая ситуация, 
когда Location иммутабельный, а вот дополнительное поле -- нет. Для решения этой проблемы можно попробовать организовать работу с иммутабельным словарём, но нужно будет его как-то инициализировать стартовыми значения,
а если заходим добавить туда новый регион, то придётся заниматься пересозданием словаря. То есть, такой подход создаёт дополнительные вопросы.

Однако, этот подход оправдан и выглядит неплохо, когда мы не можем расширить нужный класс, а новую фичу добавить нужно. 
Плюс многое зависит от контекста. Возможно нам не нужна иммутабельность полей класса, тогда моя претензия про сложность поддержки иммутабельности, становится неактуальной.
Массово я бы не стал применять такой подход, но для решения определённого типа проблем он вполне годится.
