public class LockManager {
    
    public AcquiredLock tryAcquire(long lockKey) {
        String query = "SELECT pg_try_advisory_lock(?)";
        boolean acquired;

        try (PreparedStatement ps = getConnection().prepareStatement(query)) {
            ps.setLong(1, lockKey);
            ResultSet rs = ps.executeQuery();
            rs.next();
            acquired = rs.getBoolean(1);
        }

        return acquired ? new AcquiredLock(lockKey) : null;
    }
}

public class AcquiredLock {
    private final long lockKey;
    private boolean released = false;

    public AcquiredLock(long lockKey) {
        this.lockKey = lockKey;
    }
    
    public void release() {
        if (released) {
            throw new IllegalStateException("Lock already released");
        }

        String query = "SELECT pg_advisory_unlock(?)";

        try (PreparedStatement ps = getConnection().prepareStatement(query)) {
            ps.setLong(1, lockKey);
            ps.execute();
        }
    }
}

Есть некий класс для работы с заказом. Методы этого класса следует вызывать в правильном порядке, иначе возникнут проблемы с логикой.
Новая версия содержит свой класс для каждого состояния заказа. Это защищает от вызова методов, которые недопустимы для текущего состояния. 
Пользователь должен провести заказ через всю цепочку состояний, чтобы получить доступ ко всем необходимым методам. 

### Пример 3
Было:
```java
@Setter
public class Database {
    private String url;
    private String username;
    private String password;
    private Connection connection;
    private boolean inTransaction = false;

    public void connect() {
        if (url == null) {
            throw new IllegalStateException("Url not set");  
        } 
        if (connection != null) {
            throw new IllegalStateException("Already connected");  
        } 
        this.connection = DriverManager.getConnection(url, username, password);
    }

    public void beginTransaction() {
        if (connection == null) {
            throw new IllegalStateException("Not connected");  
        } 
        if (inTransaction) {
            throw new IllegalStateException("Already in transaction");  
        } 
        connection.setAutoCommit(false);
        inTransaction = true;
    }

    public ResultSet query(String sql) {
        if (connection == null) {
            throw new IllegalStateException("Not connected");
        } 
        return connection.createStatement().executeQuery(sql);
    }

    public void commit() {
        if (!inTransaction) {
            throw new IllegalStateException("No active transaction");
        } 
        connection.commit();
        inTransaction = false;
    }

    public void rollback() {
        if (!inTransaction) {
            throw new IllegalStateException("No active transaction");
        } 
        connection.rollback();
        inTransaction = false;
    }

    public void disconnect() {
        if (connection == null) {
            throw new IllegalStateException("Not connected");
        } 
        if (inTransaction) {
            throw new IllegalStateException("Transaction still active!");
        }
        connection.close();
        connection = null;
    }
}
```
Стало:
```java
@RequiredArgsConstructor
public class DisconnectedDatabase {
    private final String url;

    public ConnectedDatabase connect(String username, String password) throws SQLException {
        Connection conn = DriverManager.getConnection(url, username, password);
        return new ConnectedDatabase(conn);
    }
}

@RequiredArgsConstructor
public class ConnectedDatabase {
    private final Connection connection;

    public ResultSet query(String sql) throws SQLException {
        return connection.createStatement().executeQuery(sql);
    }
    
    public TransactionalDatabase beginTransaction() throws SQLException {
        connection.setAutoCommit(false);
        return new TransactionalDatabase(connection);
    }

    public void disconnect() throws SQLException {
        connection.close();
    }
}

@RequiredArgsConstructor
public class TransactionalDatabase {
    private final Connection connection;

    public ResultSet query(String sql) throws SQLException {
        return connection.createStatement().executeQuery(sql);
    }

    public ConnectedDatabase commit() throws SQLException {
        connection.commit();
        connection.setAutoCommit(true);
        return new ConnectedDatabase(connection);
    }

    public ConnectedDatabase rollback() throws SQLException {
        connection.rollback();
        connection.setAutoCommit(true);
        return new ConnectedDatabase(connection);
    }
}
```

Есть один универсальный класс для работы с БД, в котором важно вызывать методы в правильном порядке, иначе будем получать ошибки. 
В новой версии кода создано три класса, которые отражают разное состояние БД: есть соединение с БД, нет соединения, БД в режиме транзакции. 
При изменении состояния мы получаем новый объект. У этих классов отличаются наборы методов.
Например, теперь мы не можем откатить коммит, если у нас нет подключения к БД.


#### Вывод
После этого занятия, даже стало немного жаль, что в Java нет вещей, которые тут были описаны.  
Узнал, что такое Алгебрические типы данныx :)   
TOP довольно интересная вещь, благодаря такому подходу мы можем описывать состояния объекта и допустимые для этого состояния действия. Код становится менее подверженным ошибкам.
К сожалению, в популярных языках современности с поддержкой TOP туго и приходится идти на всякие ухищрения. Можно попробовать применить паттерн state или state machine, 
но у меня возникает такое ощущение, что это лишь отдалённое подобие TOP, которое не очень удобно и имеет свои недостатки. Например, при использовании паттерна State нам будет трудно гарантировать, 
что клиент не будет использовать операции, недопустимые для конкретного состояния. Впрочем, даже в таких ограниченных вариантах мы можем получить более безопасный код.
Но слишком фанатично относиться к работе с состоянием тоже вредно, так как иногда проще будет показать состояние булевым флагом, а не создавать иерархию классов.
