CommandInfo getFullCommandName(String uri) {
    //Когда ответ пуст? Когда uri пуст
    if (StringUtils.isBlank(uri)) {
        return new CommandInfo("", "");
    }

    String[] parts = uri.split("/");

    //Получить имя сущности
    String entityName = getEntityName(parts);

    //Получить имя команды
    String commandName = getCommandName(parts);

    //Собрать всё в один объект
    return new CommandInfo(entityName, commandName);
}

private String getEntityName(String[] uriParts) {
    return uriParts.length >= 2 ? uriParts[uriParts.length - 2] : "";
}

private String getCommandName(String[] uriParts) {
    return uriParts.length >= 2 ? uriParts[uriParts.length - 1] : "";
}

Имеется метод, который извлекает из uri запроса название вызываемой сущности и команды. Изначальная версия сосредоточена на входных данных.

#### Вывод
При выборе решения нужно смотреть не только на структуру входных данных, но и на структуру выходных данных.
Возможно взгляд со стороны выходных данных натолкнёт на более аккуратное решение.
Ко-рекурсия интересный подход, который может упростить жизнь в некоторых случаях, но иногда проще работать через привычную структурную рекурсию.  
У меня в голове сложилась такая иерархия подходов:  
`Структурная рекурсия -> ко-рекурсия -> генеративная рекурсия`
Если трудно обойтись структурной рекурсией, то стоит присмотреться к ко-рекурсии и т.д.
Ещё немного смущало название рекурсивный и ко-рекурсивный подход. Мы же способны применять их и для составных структур данных(не список и не дерево, а обычный объект).
Но, насколько я понял, главный посыл этого занятия -- смотреть на программу не только со стороны входных данных. А ко-рекурсия лишь один из подходов, в этом деле.  

Было трудно найти хорошие примеры. Полноценную рекурсию я не видел давно, обычно всё решается в итеративном/императивном стиле.  
Интересно это особенности моих проектов или языка? Почувствовал себя жертвой императивщины, которой тяжело соображать в формате рекурсии.  

Стало интересно почему императивные фукнкции не задают дизайн программы(об этом было написано в материале).
Задавался вопросом, а работает ли такой подход с итеративным стилем, который я вижу гораздо чаще. Показалось, что не работает, так как в итеративном/императивном подходе мы сосредоточены на том что делать. Интересно, потому так не любят императивщину, что она отвлекает от главного: данных? 
Кажется, что в императивном подходе фокус внимания смещён на входные данные, так как нам нужно задавать способ их обработки. 
Хотя тут ещё всё зависит от того, с чем мы конкретно работаем. Если проходиться по коллекции в цикле, то ко-рекурсию организовать сложно. 
Но при создании какого-то сложного объекта с множеством полей, вполне можно её применить. Написал это и подумал, что путаю понятия. Мы же можем писать на Java в функциональном стиле. 
И выбор языка не привязывает нас строго к одному стилю программирования, может принуждает или ограничивает, но всё равно у нас остаются варианты принести частицы того или иного стиля в используемый язык.  
А значит главный вывод всё же можно сделать такой.   
Решаешь проблему? Задумайся: а не проще ли подойти к ней со стороны выходных данных.

Кажется, что в ряде случае получается более приятный код.
